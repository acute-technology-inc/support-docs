# I80 (Intel 8080 Parallel Bus)

## What is I80?

### Overview

I80, also known as the Intel 8080 parallel bus interface or 8080 MCU interface, is a parallel communication protocol derived from the Intel 8080 microprocessor's bus architecture, widely adopted as a standard interface for LCD display controllers, TFT displays, OLED panels, and other memory-mapped peripheral devices. Despite its origins in the 1970s 8080 CPU, the I80 bus protocol has evolved into a de facto standard for display module connectivity in embedded systems, particularly for small to medium-sized TFT-LCD and OLED displays used in consumer electronics, industrial controls, and medical devices. The interface provides a simple, efficient method for microcontrollers and SoCs to send commands and data to display controllers without requiring dedicated display hardware.

The I80 interface uses a parallel data bus (typically 8-bit or 16-bit) along with control signals including CS (Chip Select), RS/DC (Register Select/Data Command), WR (Write), RD (Read), and RESET. Data transfer occurs through asynchronous read and write cycles initiated by the host controller, with timing parameters defined by setup, hold, and pulse width specifications rather than a synchronous clock. The RS/DC signal distinguishes between command writes (register access, configuration) and data writes (pixel data, framebuffer), allowing the display controller to interpret incoming bytes appropriately. This simple control mechanism makes I80 easy to implement with GPIO pins on virtually any microcontroller, though dedicated parallel interface peripherals can achieve higher throughput.

I80 interfaces are ubiquitous in display modules for portable devices, industrial HMIs (Human-Machine Interfaces), automotive instrument clusters, and IoT products. Common display controllers supporting I80 include ILI9341, ST7789, SSD1306, ILI9486, and many others from manufacturers like Ilitek, Sitronix, Solomon Systech, and Raydium. The interface typically operates at effective speeds of 1-20 MHz equivalent transfer rates, balancing performance with the GPIO capabilities of common microcontrollers. While newer displays increasingly adopt serial interfaces (SPI, MIPI DSI) for reduced pin count, I80 remains prevalent due to its simplicity, legacy support, and superior throughput compared to SPI for larger displays.

### Key Features

- **Parallel Data Bus**: 8-bit or 16-bit wide data transfer
- **Asynchronous Operation**: No clock signal, timing based on control strobes
- **Simple Control**: CS, RS/DC, WR, RD, RESET signals
- **Command/Data Mode**: RS/DC distinguishes commands from data
- **Bidirectional**: Supports both read and write operations
- **Memory-Mapped**: Direct access to display controller registers
- **GPIO Compatible**: Can be bit-banged on any microcontroller
- **Fast Transfers**: Higher throughput than SPI for equivalent clock speeds
- **Industry Standard**: Supported by most LCD/OLED display controllers
- **Flexible Timing**: Configurable to match various controller specs

## Technical Specifications

### Signal Description

**Control Signals**

- **CS (Chip Select)**: Active low, enables display controller
- **RS / DC / D/C (Register Select / Data Command)**: 
  - Low = Command (register address, configuration)
  - High = Data (pixel data, parameters)
- **WR / WRX (Write)**: Active low, latches data on rising edge
- **RD / RDX (Read)**: Active low, enables data output (optional, read operations)
- **RESET / RST**: Active low, hardware reset of display controller

**Data Bus**

- **D[7:0]**: 8-bit parallel data bus (most common)
- **D[15:0]**: 16-bit parallel data bus (faster variants, often split as D[15:8] for high byte, D[7:0] for low byte)

**Optional Signals**

- **VSYNC / HSYNC**: Synchronization signals (some controllers)
- **TE (Tearing Effect)**: Frame synchronization output from display
- **Backlight Control**: PWM or enable signal (separate from I80 bus)

**Electrical Characteristics**
- **Voltage Levels**: 3.3V or 1.8V CMOS (controller-dependent)
- **Current**: Standard CMOS drive strength (2-8 mA per pin)
- **Speed**: Typically DC to 20 MHz equivalent write cycle rate

### Timing Specifications

**Write Cycle Timing**

Typical timing parameters (vary by display controller):

- **CS Setup Time (tCSS)**: 10-50 ns (CS asserted before WR)
- **CS Hold Time (tCSH)**: 10-50 ns (CS held after WR)
- **RS Setup Time (tRSS)**: 10-50 ns (RS stable before WR)
- **RS Hold Time (tRSH)**: 10-50 ns (RS held after WR)
- **Data Setup Time (tDS)**: 10-50 ns (data stable before WR rising)
- **Data Hold Time (tDH)**: 10-50 ns (data held after WR rising)
- **WR Pulse Width Low (tWRL)**: 15-100 ns (WR must be low)
- **WR Pulse Width High (tWRH)**: 15-100 ns (WR must be high)
- **Write Cycle Time (tWC)**: 50-200 ns (minimum time between writes)

**Read Cycle Timing**

- **RD Pulse Width Low (tRDL)**: 100-300 ns (RD asserted)
- **RD to Data Valid (tRDD)**: 50-200 ns (data output delay)
- **RD to Data Hold (tRDH)**: 10-50 ns (data hold after RD rising)
- **Read Cycle Time (tRC)**: 150-400 ns (minimum time between reads)

**Example Fast Timing** (ILI9341 at 3.3V):
- WR cycle: ~66 ns minimum (15 MHz write rate)
- Typical practical: 100-200 ns (5-10 MHz)

### Write Transaction Sequence

**Command Write** (e.g., set cursor position)

```
1. Assert CS (low)
2. Set RS/DC = 0 (command mode)
3. Place command byte on D[7:0]
4. Assert WR (low): setup time
5. Deassert WR (high): command latched on rising edge
6. Hold data and RS - hold time
7. Deassert CS (high) or continue with data
```

**Data Write** (e.g., write pixel data)

```
1. Assert CS (low) [may already be asserted from command]
2. Set RS/DC = 1 (data mode)
3. Place data byte/word on D[15:0] or D[7:0]
4. Assert WR (low)
5. Deassert WR (high): data latched
6. Hold data and RS
7. Repeat steps 3-6 for multiple pixels
8. Deassert CS when done
```

### Common Display Controller Commands

**ILI9341 / ST7789 Examples** (typical LCD controller)

- **0x01**: Software Reset
- **0x11**: Sleep Out
- **0x28**: Display OFF
- **0x29**: Display ON
- **0x2A**: Column Address Set (X start/end)
- **0x2B**: Page Address Set (Y start/end)
- **0x2C**: Memory Write (followed by pixel data)
- **0x36**: Memory Access Control (rotation, mirroring)
- **0x3A**: Pixel Format Set (RGB565, RGB666, etc.)

**Command-Parameter Sequence**

1. Write command byte (RS=0)
2. Write parameter byte(s) (RS=1)
3. Repeat for more parameters
4. Write data bytes if applicable (RS=1)

### Data Formats

**Pixel Formats**

- **RGB565**: 16-bit color (5 bits red, 6 green, 5 blue): most common
- **RGB666**: 18-bit color packed in 3 bytes
- **RGB888**: 24-bit true color (3 bytes per pixel)
- **Indexed Color**: 8-bit or less with palette

**Byte Order** (for RGB565, 16-bit bus)

- **Big Endian**: High byte first [R4R3R2R1R0G5G4G3], then [G2G1G0B4B3B2B1B0]
- **Little Endian**: Low byte first (less common in display controllers)

## Decoder Configuration

When analyzing I80 bus traffic with a logic analyzer, configure the following parameters:

### Signal Connections

**Minimum Configuration** (write-only, 8-bit)
- CS - Chip select (1 channel)
- RS/DC - Register select / Data-command (1 channel)
- WR - Write strobe (1 channel)
- D[7:0] - 8-bit data bus (8 channels)
- **Total**: 11 channels

**With Read and 16-bit** (full configuration)
- Add RD - Read strobe (1 channel)
- Add D[15:8] - Upper data byte (8 channels)
- Add RESET - Reset signal (1 channel, optional)
- **Total**: 21 channels

### Sampling Requirements

- **Minimum Sample Rate**: 100 MS/s for typical displays
- For fast controllers (15-20 MHz write rate): 200-500 MS/s recommended
- For slow controllers (1-5 MHz): 50-100 MS/s sufficient

### Decoder Parameters

- **Bus Width**: 8-bit or 16-bit
- **Control Signal Polarity**: CS active low, WR active low, RD active low (typical)
- **RS/DC Polarity**: 0 = Command, 1 = Data (standard)
- **Display Controller**: Specify controller type (ILI9341, ST7789, etc.) for command decoding
- **Pixel Format**: RGB565, RGB666, RGB888 (for data interpretation)

### Display Options

- Show transactions as Command or Data based on RS/DC
- Decode known commands (e.g., 0x2A = Column Address Set)
- Display data values in hex or as pixel colors
- Annotate command parameters
- Group command + parameter sequences
- Show timing measurements (pulse widths, cycle times)

### Trigger Settings

- Trigger on CS falling edge (start of transaction)
- Trigger on specific command code (e.g., 0x2C for pixel write)
- Trigger on RS/DC transition (command to data)
- Trigger on WR pulse with specific data pattern
- Trigger on RESET signal

### Analysis Tips

1. **Identify Initialization Sequence**
   - Capture power-on or reset
   - Look for series of commands configuring display
   - Common: Reset, Sleep Out, Display On, Pixel Format, Memory Access

2. **Decode Commands**
   - RS/DC = 0 indicates command byte
   - Following bytes with RS/DC = 1 are parameters or data
   - Consult display controller datasheet for command list

3. **Verify Timing**
   - Measure WR pulse width (must meet tWRL, tWRH)
   - Check setup and hold times for data and RS
   - Verify cycle time between consecutive writes

4. **Analyze Pixel Data**
   - After 0x2C (Memory Write) command, data stream is pixels
   - Decode RGB565 to visualize colors
   - Calculate framebuffer fill rate

5. **Check for Timing Violations**
   - Data changing during WR low (setup violation)
   - Data not held after WR rising (hold violation)
   - WR pulses too fast for controller specs

6. **Monitor CS Usage**
   - CS can stay asserted for burst writes (faster)
   - CS deasserted between transactions (slower, safer)
   - Check for proper CS timing margins

7. **Observe Frame Updates**
   - Identify frame start (usually 0x2A, 0x2B, 0x2C sequence)
   - Measure frame update time
   - Calculate FPS (frames per second)

### Common Issues

**Display Not Initializing**
- **Symptom**: Blank screen, no response
- **Cause**: Missing reset, wrong command sequence, timing violations
- **Solution**: Verify RESET pulse, check initialization commands, slow down timing

**Corrupted Display**
- **Symptom**: Wrong colors, garbled pixels, shifted image
- **Cause**: Incorrect pixel format, wrong address range, bus errors
- **Solution**: Verify RGB format setting (0x3A command), check column/page addresses

**Flickering or Tearing**
- **Symptom**: Visual artifacts during screen updates
- **Cause**: Writing to framebuffer during scan-out, no VSYNC sync
- **Solution**: Use TE signal for synchronization, implement double buffering

**Timing Violations**
- **Symptom**: Intermittent errors, missing writes
- **Cause**: Setup/hold times not met, WR pulse too narrow
- **Solution**: Increase delays, check microcontroller GPIO speed limits

**Wrong Data Width**
- **Symptom**: Every other pixel wrong, or half of display corrupted
- **Cause**: Using 8-bit when controller expects 16-bit or vice versa
- **Solution**: Verify bus width configuration, check D[15:8] connections

**CS Not Asserted**
- **Symptom**: Controller not responding to any commands
- **Cause**: CS stuck high or not connected
- **Solution**: Verify CS wiring and GPIO configuration

### Advanced Analysis

**Performance Measurement**
- Calculate effective pixel write rate
- Measure framebuffer update time for full screen
- Identify bottlenecks (CPU speed, timing delays)
- Optimize write burst lengths

**Protocol Compliance**
- Verify all timing parameters against datasheet
- Check setup, hold, pulse width specifications
- Validate command sequences
- Test edge cases (minimum/maximum timing)

**Display Controller Debugging**
- Read status registers (if RD implemented)
- Verify controller ID registers
- Check configuration registers match expected
- Monitor internal state transitions

**Pixel Data Visualization**
- Decode RGB565/RGB666/RGB888 to actual colors
- Reconstruct framebuffer from write sequence
- Visualize image being drawn
- Identify drawing patterns (lines, rectangles, text)
