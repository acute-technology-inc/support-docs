# QSPI

## What is QSPI?

QSPI (Quad Serial Peripheral Interface) is an enhanced version of the standard SPI protocol that uses four bidirectional data lines instead of the two unidirectional lines (MOSI/MISO) in traditional SPI. This quad-line architecture enables four bits to be transferred simultaneously per clock cycle, quadrupling the potential throughput compared to standard SPI while maintaining the familiar SPI command structure and chip select mechanism. QSPI was developed primarily for high-speed communication with NOR Flash memory devices, where fast sequential reads and random access performance are critical for applications like code execution-in-place (XIP) and fast boot times.

The protocol extends the traditional SPI four-wire interface (CLK, CS#, MOSI, MISO) into a six-wire interface by transforming the traditional data lines into four bidirectional I/O lines (IO0, IO1, IO2, IO3). During different phases of a transaction, these lines can operate in single, dual, or quad modes depending on the command and device capabilities. This flexibility allows QSPI to maintain backward compatibility with standard SPI while providing the option for dramatically increased bandwidth when both controller and device support quad mode. Modern QSPI implementations can achieve sustained sequential read rates approaching 100 MB/s and support features like DDR (Double Data Rate) operation for even higher performance.

QSPI has become the standard interface for Serial NOR Flash memory in embedded systems, microcontrollers, FPGAs, and system-on-chip (SoC) designs where fast, non-volatile storage is required. The protocol is particularly valuable for storing firmware, boot code, configuration data, and frequently accessed application code. While not formally standardized by a single governing body, QSPI follows conventions established by major Flash memory manufacturers and is supported by virtually all microcontroller vendors. The JEDEC Serial Flash Discoverable Parameters (SFDP) standard provides a mechanism for Flash devices to advertise their QSPI capabilities, enabling universal QSPI controller implementations.

## Technical Specifications

### Signal Lines

**Six-Wire QSPI Interface:**
- **CLK (Clock)**: Serial clock generated by master
- **CS# (Chip Select)**: Active-low device select
- **IO0**: Bidirectional data line 0 (MOSI in standard SPI)
- **IO1**: Bidirectional data line 1 (MISO in standard SPI)
- **IO2**: Bidirectional data line 2 (typically WP#/Write Protect in standard SPI)
- **IO3**: Bidirectional data line 3 (typically HOLD#/Reset in standard SPI)

**Note**: IO2 and IO3 serve dual purposes. In standard SPI mode, they function as Write Protect and Hold/Reset control signals. In quad mode, they become data lines.

### Operational Modes

QSPI supports multiple transfer modes, typically denoted as x-y-z format:
- **x**: Number of data lines for command phase
- **y**: Number of data lines for address phase
- **z**: Number of data lines for data phase

**Common Modes:**
- **1-1-1 (Standard SPI)**: Single data line throughout - backward compatible
- **1-1-2 (Dual Output)**: Single for command/address, dual for data read
- **1-2-2 (Dual I/O)**: Single for command, dual for address and data
- **1-1-4 (Quad Output)**: Single for command/address, quad for data read
- **1-4-4 (Quad I/O)**: Single for command, quad for address and data
- **4-4-4 (QPI)**: Quad throughout all phases (requires mode switch command)

**DDR Variants:**
- Many modes also support DDR (Double Data Rate) where data transfers on both rising and falling clock edges, effectively doubling throughput

### Data Rates

**Clock Frequencies:**
- Standard SPI mode: 1-133 MHz typical
- Quad mode: 80-133 MHz typical (device-dependent)
- High-performance devices: Up to 166 MHz

**Effective Bandwidth:**
- Standard SPI @ 133 MHz: ~16.6 MB/s
- Quad I/O @ 133 MHz: ~66.5 MB/s (4× parallelism)
- Quad DDR @ 133 MHz: ~133 MB/s (4× + DDR)

Actual performance depends on command overhead and device-specific limitations.

### Command Structure

**Typical QSPI Read Transaction:**
1. Assert CS# (low)
2. Send command byte (e.g., 0xEB for Quad I/O Read) on IO0
3. Send address bytes (3 or 4 bytes) on IO0-3 (quad mode)
4. Send mode/dummy bytes if required
5. Receive data bytes on IO0-3 (quad mode)
6. De-assert CS# (high)

**Common QSPI Commands:**
- **0x03**: Standard Read (1-1-1)
- **0x0B**: Fast Read with dummy cycles
- **0x3B**: Dual Output Fast Read (1-1-2)
- **0xBB**: Dual I/O Fast Read (1-2-2)
- **0x6B**: Quad Output Fast Read (1-1-4)
- **0xEB**: Quad I/O Fast Read (1-4-4)
- **0x38/0xFF**: Enter/Exit QPI mode (4-4-4)

**Note**: Command sets vary between manufacturers. Always refer to device datasheets.

### Addressing

**3-Byte Addressing:**
- Traditional mode supporting up to 16 MB (128 Mbit)
- Address sent as 3 bytes (24 bits)

**4-Byte Addressing:**
- Extended mode for devices >16 MB (up to 4 GB theoretically)
- Address sent as 4 bytes (32 bits)
- May require mode switching command

### Dummy Cycles

Many QSPI commands require dummy cycles between address and data phases:
- Allows device internal circuitry time to prepare data
- Number of dummy cycles varies by command and clock frequency
- Configurable in some devices to optimize for different speeds

### Execute-in-Place (XIP)

Some QSPI controllers support XIP mode:
- Memory-mapped access to Flash contents
- Controller automatically issues read commands as CPU accesses memory addresses
- Enables code execution directly from Flash without copying to RAM
- Reduces RAM requirements and accelerates boot time

## Common Applications

QSPI is used extensively for non-volatile storage in embedded systems:

- **Microcontroller firmware storage**: Boot code and application firmware
- **FPGA configuration**: Bitstream storage for FPGA programming
- **SoC boot ROM**: System initialization code for complex SoCs
- **Embedded Linux**: Kernel, root filesystem, and data storage
- **IoT devices**: Firmware and data logging in connected devices
- **Automotive ECUs**: Software updates and calibration data
- **Industrial controllers**: PLC programs and configuration
- **Network equipment**: Router/switch firmware and configuration
- **Consumer electronics**: Smart appliances, set-top boxes
- **Medical devices**: Embedded firmware in diagnostic equipment
- **Wearables**: Firmware storage in smartwatches and fitness trackers
- **Drones**: Flight controller firmware and maps
- **Gaming**: Firmware for controllers and peripherals
- **Audio/video equipment**: Firmware for DSPs and processors
- **Single-board computers**: Boot storage for SBCs and modules

## Decoder Configuration

When configuring a logic analyzer to decode QSPI signals:

### Channel Assignment

**Minimum Required (Standard SPI mode):**
- **CLK**: Clock
- **CS#**: Chip Select
- **IO0** (MOSI): Data output
- **IO1** (MISO): Data input

**Full QSPI (Quad mode):**
- **CLK**: Clock (required)
- **CS#**: Chip Select (required)
- **IO0**: Data line 0 (required)
- **IO1**: Data line 1 (required)
- **IO2**: Data line 2 (required for quad modes)
- **IO3**: Data line 3 (required for quad modes)

### Protocol Parameters

- **SPI Mode**: Mode 0 (CPOL=0, CPHA=0) or Mode 3 (CPOL=1, CPHA=1): most QSPI uses Mode 0 or 3
- **Transfer modes**: Select supported modes (1-1-1, 1-1-4, 1-4-4, 4-4-4, etc.)
- **Clock frequency**: Set expected clock rate
- **Address size**: 3-byte or 4-byte addressing
- **Command set**: Load device-specific command definitions if available

### Decoding Options

- **Command identification**: Display command names (Read, Fast Read, Quad Read, etc.)
- **Mode display**: Show current transfer mode (1-1-1, 1-4-4, etc.)
- **Address display**: Show target address in hex
- **Data payload**: Display read/write data in hex/ASCII
- **Dummy cycle indication**: Mark dummy clock cycles
- **Transfer direction**: Indicate read vs. write operations

### Trigger Configuration

- **CS# assertion**: Trigger on chip select going low (transaction start)
- **Specific command**: Trigger on specific command byte (e.g., Quad Read)
- **Address match**: Trigger on access to specific memory address
- **Mode change**: Trigger on QPI mode entry/exit commands

### Sampling Requirements

**Minimum Sampling Rate:**
- At least 4× the clock frequency
- Example: 133 MHz clock requires 532 MHz sampling minimum

**Recommended Sampling Rate:**
- 10× clock frequency for detailed analysis
- Example: 133 MHz clock requires 1.33 GHz sampling

**Buffer Depth:**
- Sequential reads can be megabytes long
- Configure appropriate buffer or use segmented capture

### Analysis Tips

When analyzing QSPI communications:

1. **Start with standard SPI**: Capture initialization and mode changes at slower speeds first
2. **Identify command phase**: Commands are always sent on IO0 in most modes
3. **Count dummy cycles**: Verify correct dummy cycle count for chosen command
4. **Track mode switches**: Watch for commands that change transfer mode (e.g., Enter QPI)
5. **Verify data integrity**: Cross-check read data against known values if possible
6. **Observe CS# management**: Some controllers keep CS# low for multiple commands (continuous mode)
7. **Monitor signal integrity**: High-speed quad mode is sensitive to signal quality
8. **Check for XIP patterns**: Repeated reads with sequential addresses indicate XIP operation

### Common Protocol Sequences

**Device Initialization:**
1. Read device ID (command 0x9F, 1-1-1 mode)
2. Read SFDP to discover capabilities
3. Configure device (set dummy cycles, enable quad mode if needed)
4. Optionally enter QPI mode (4-4-4)

**Quad Read Sequence:**
1. CS# asserted (low)
2. Command 0xEB on IO0 (1-bit)
3. Address on IO0-3 (4-bit, 3 or 4 bytes)
4. Mode byte on IO0-3 (4-bit)
5. Dummy cycles (typically 4-6)
6. Data on IO0-3 (4-bit, multiple bytes)
7. CS# de-asserted (high)

**QPI Mode Operation:**
After entering QPI with command 0x38, all subsequent transactions use 4-4-4 mode including commands and addresses, maximizing throughput.
